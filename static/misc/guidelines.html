Guidelines for this website

2025-08-17

A website is a complicated thing. There's a huge amount of stuff to consider, decades of legacy cruft, and a lot to avoid (as they say, “it's the notes you don't play.”)

This page is written largely for myself, so that I don't lose track of my own goals and methods for this site, but it's reasonable to record these in a public place in case anyone like me (i.e., bad at this) wants some resources to help them write good, accessible websites.

Now, I'll note that I'm pretty stupid when it comes to frontend stuff, and that creates some amount of incentive for me to keep things simple. So I have to confess that some of this might be rationalization: It's not that I'm making websites the only way I know how, it's that I'm making websites the better way!

But also, I seem to have different design goals than many others. I want a website that works without javascript wherever I don't need it. I want a website that's reasonably functional without CSS. I want a website that's mostly HTML, and doesn't try to reimplement stuff the browser does ten times better and faster, and with better accessibility characteristics. I want a website where most of the data you download is actual content that you see.

Accessibility is a priority — probably the priority. That means accessibility for people with slow or intermittent internet, slow devices, old and underpowered browsers, or small screens. It also means accessibility for people who use screen readers or other assistive technology. Generally, these benefits compound and help everybody; a website that has these characteristics is a website that's fast for those with good internet, is understandable for search engines and reasonably easy for machine translation, and can be used by people working without images in a terminal.

It's also worth paying attention to the different ways people want to use the web. Respecting dark- and light-mode settings with media queries is the most obvious example. Some users have more “heavy-weight” opinions, though. For example, there are those with strong opinions on javascript.

There are also those who, like me, often “print” pages from the web. For me, this is to download onto an e-ink device. Supporting this well involves using media queries for printing. One straightforward choice is to display link targets for printed content, since you can't exactly click a link on the web.

I will admit this site doesn't do things perfectly in terms of accessibility right now. While I probably don't need to bother with aria, I ought to better follow the MDN's accessibility recommendations. There's also the matter that many semantic elements are (relatively) newfangled; how portable is HTML5 and CSS3 (and SVG support)? Is it a good tradeoff to add polyfills for browsers which don't quite handle semantic HTML?

Language and encoding are also considerations, since the web is a global platform and “English is the implicit default” isn't actually a good way for the world to work. As part of that, it's good to declare which language you're writing in, so that the browser and OS don't have to guess, and that you're using UTF-8, since that's basically what you have to use.
